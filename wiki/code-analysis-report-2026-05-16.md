  2. КРИТИЧНІ БАГИ (CRITICAL)
═══════════════════════════════════════════════════════════════

  [B1] lib/_core/manus-runtime.ts — Дублювання функцій (Рядки 72, 99)
       ─────────────────────────────────────────────────────────
       Файл містить ДВІ визначення handleMessage (рядки 72-90 і 99-115)
       та ДВІ визначення subscribeSafeAreaInsets (рядки 92-97 і 120-127).
       Другі визначення посилаються на змінну safeAreaCallback (однина),
       яка НІДЕ не оголошена у файлі. Це призведе до ReferenceError
       під час виконання. initManusRuntime додає слухач повідомлень,
       який використовує handleMessage — і через JS hoisting перемагає
       друге визначення, що викличе crash.

       Виправлення: Видалити дублікати (рядки 99-127), залишити Set-
       підхід або оголосити safeAreaCallback як module-level variable.

  [B2] components/effects/SunraysEffect.tsx — Невизначена змінна (Рядок 67)
       ─────────────────────────────────────────────────────────
       Змінна rayCount використовується, але ніколи не оголошена.
       Мала бути count (оголошена на рядку 43). Runtime crash при
       спробі відобразити ефект сонячних променів.

       Виправлення: Замінити rayCount на count.

  [B3] app/editor-blocks.tsx — Відсутній імпорт (Рядок 20)
       ─────────────────────────────────────────────────────────
       useRouter використовується, але ніколи не імпортується.
       Runtime crash при навігації на сторінку editor-blocks.

       Виправлення: Додати import { useRouter } from 'expo-router'.

  [B4] components/effects/FogEffect.tsx — Зламаний useEffect (Рядки 41-47)
       ─────────────────────────────────────────────────────────
       useEffect створює новий масив layers з Animated.Value об'єктами
       при кожному запуску ефекту, але змінна layers перекриває
       layers з useMemo вище. Cleanup зупиняє анімації тільки з
       inner scope, а старі Animated.Value об'єкти витікають.
       Це призводить до memory leak та візуальних глюків.

  [B5] stores/use-app-store.ts — Невизначена змінна charLib (Рядок 157)
       ─────────────────────────────────────────────────────────
       characterLibraries: typeof charLib === 'object' && !Array.isArray(charLib) ? charLib : {}
       Змінна charLib ніколи не оголошена у функції migrateFromLegacyKeys.
       Мала бути characterLibraries з попереднього блоку try/catch.
       Це призведе до ReferenceError при міграції старих даних.

  3. ВАЖЛИВІ ПРОБЛЕМИ (HIGH)
═══════════════════════════════════════════════════════════════

  [H1] lib/character-animator.ts — Hardcoded screenWidth (Рядок 219)
       ─────────────────────────────────────────────────────────
       createHideAnimation задає screenWidth = 500 захардкожено,
       тоді як createCharacterAnimation приймає screenWidth як
       параметр. Анімації вильатимуть за екран на пристроях з
       шириною відмінною від 500px.

  [H2] lib/audio-trigger-scheduler.ts — Не резолвить проміси (Рядки 109-114)
       ─────────────────────────────────────────────────────────
       cancelAllTriggers очищає timeouts, але не викликає resolve()
       на промісах, на відміну від cancelTrigger. Будь-який
       await на тригері зависне назавжди → memory leak.

  [H3] lib/story-context-enhanced.ts — Без обмеження розміру JSON (Рядок 44-58)
       ─────────────────────────────────────────────────────────
       importStory приймає довільний рядок JSON без перевірки
       розміру. Зловмисник може передати гігабайтний JSON,
       що призведе до OOM. StoryValidator.validateStory перевіряє
       довжину тексту (50000 символів), але не загальний розмір
       історії або кількість сцен.

  [H4] lib/character-library.ts —Мутація масивів Zustand (Рядки 33, 71)
       ─────────────────────────────────────────────────────────
       addCharacter виконує library.push(newCharacter), а
       addSpriteToCharacter — character.sprites.push(newSprite).
       Zustand очікує immutable updates. Мутація масиву на місці
       може не викликати re-render, оскільки референс масиву
       не змінюється.

  [H5] lib/story-domain.ts — sceneTitle отримує sceneId (Рядок 33)
       ─────────────────────────────────────────────────────────
       const sceneTitle = currentScene?.id;
       Змінна називається sceneTitle, але містить id. Слоти
       збережень показуватимуть внутрішні ID замість читабельних
       назв.

  [H6] lib/_core/manus-runtime.ts — postMessage з wildcard origin (Рядок 56)
       ─────────────────────────────────────────────────────────
       window.parent.postMessage(message, "*") — будь-який origin
       може отримати повідомлення. Також немає валідації
       event.origin у вхідних повідомленнях.

  [H7] components/effects/SparklesEffect.tsx — Дублікат cleanup (Рядки 88-95)
       ─────────────────────────────────────────────────────────
       Два return () => cleanup блоки в useEffect. Другий
       (рядки 93-95) — недосяжний мертвий код.

  [H8] app/scene-editor.tsx — Мертвий код FilePickerRow (Рядки 46-103)
       ─────────────────────────────────────────────────────────
       Локальний компонент FilePickerRow визначений, але ніколи
       не використовується. SceneEditorForm використовує
       MediaPickerRow через контекст.
