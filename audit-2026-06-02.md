## Top Critical Findings

### CR-1: PII / session-token leak to dev console in OAuth callback
- **File:** `app/oauth/callback.tsx:46`
- **Category:** Security
- **Severity:** CRITICAL
- **Description:** `console.log("[OAuth] User info fetched from API:", authUserInfo)` dumps the entire `User` object (id, openId, **email**, name, loginMethod) to console in __DEV__. Although guarded by `__DEV__`, the user's email/PII should never be in a console call. The `__DEV__` flag is **always true in dev builds** shipped to QA/internal testers.
- **Fix effort:** S
- **Suggested fix:** Replace object with `console.log("[OAuth] User fetched:", { id: authUserInfo.id, openId: authUserInfo.openId })` or just `console.log("[OAuth] User info fetched")`.

### CR-2: Session-token partial-leak in API log
- **File:** `lib/_core/api.ts:153-157`
- **Category:** Security
- **Severity:** CRITICAL
- **Description:** `exchangeOAuthCode` logs `sessionToken: sessionToken ? \`${sessionToken.substring(0, 50)}...\` : null`. Even first 50 chars of a session token in dev logs is excessive (token entropy is reduced; remaining char set is small). Same risk on `establishSession` (line 194).
- **Fix effort:** S
- **Suggested fix:** Log only `hasSessionToken: !!sessionToken`; never log any prefix of the token.

### CR-3: Unguarded `console.*` in 12 production files
- **Files (non-exhaustive):** `app/tabs/index.tsx:175,183,210,229,244,249`; `app/reader.tsx:64`; `lib/engine/useSceneExecutor.ts:38,204,269`; `lib/editor/story-manuscript-save.ts:77,84`; `stores/use-app-store.ts:460,463`; `hooks/useReaderAudio.ts:194`; `lib/audio-player-service.ts:30` (in logDebug — actually **guarded** on line 29 ✓); `lib/i18n.ts:26` (guarded ✓)
- **Category:** Code-Quality / Security (info leakage in shipped dev builds)
- **Severity:** CRITICAL
- **Description:** Phase 13 fixed `lib/_core/api.ts` but left these unguarded. In Expo dev builds, `__DEV__` is true and logs ship in JS bundle. Some contain scene state, PII hints, or partial tokens. `lib/editor/story-manuscript-save.ts:77,84` is the most concerning (saves happen on every scene).
- **Fix effort:** S (mechanical wrap)
- **Suggested fix:** Wrap each call: `if (__DEV__) console.log(...)`. ESLint rule `no-console` with auto-fix already exists in most Expo starters.

### WR-1: `setTimeout` without cleanup in two save handlers
- **Files:** `components/editor/SceneComposer.tsx:100`; `components/document-editor/DocumentSceneEditor.tsx:138`
- **Category:** Code-Quality (potential state-update-after-unmount warning)
- **Severity:** WARNING
- **Description:** Both call `setTimeout(() => setIsSaving(false), 500/250)` without storing the ID or returning cleanup. If the user navigates away within 500ms, React fires "state update on unmounted component".
- **Fix effort:** S
- **Suggested fix:** Store timeout ID in a ref + clear in `useEffect` cleanup; or move save state into a ref.

### WR-3: `useReaderAudio` re-initializes / re-stops on every audioManager prop change
- **File:** `hooks/useReaderAudio.ts:298-312`
- **Category:** Code-Quality (audio bug)
- **Severity:** WARNING
- **Description:** Two `useEffect`s depend on `[audioManager]` only. Every parent re-render that passes a new `audioManager` object reference causes: (1) `useEffect:298-304` to call `stopReaderPlayback(audioManager)` — interrupting playback; (2) `useEffect:306-312` to call `audioManager.initialize()` again. Also, `useFocusEffect` at 241-250 has the same dep. The hook is being passed `enhancedAudioManager as defaultAudioManager` (singleton), but the de-facto pattern is fragile.
- **Fix effort:** M
- **Suggested fix:** Drop `audioManager` from dep arrays (it's a module-level singleton) and guard `initialize()` with a module-level `initPromise` singleton. Use `useRef` for the manager if it must be a prop.

### WR-4: Hex-alpha hack back in 7 files
- **Files:** `components/WebSidebar.tsx:87` (`+ '15'`); `components/editor/SceneComposerPhone.tsx:140` (`${colors.primary}12`); `components/editor/SceneSelector.tsx:338,425` (`+ '20'`, `+ '15'`); `components/editor/MiniPreview.tsx:96,161` (`+ '30'`, `+ '20'`); `components/editor/PreviewScreen.tsx:170` (`+ '30'`); `components/editor/TimelinePanel.tsx:191` (`+ '40'`); `components/editor/modals/SaveSceneDialog.tsx:165` (`+ '20'`)
- **Category:** UI-UX (theming)
- **Severity:** WARNING
- **Description:** M2 fixed 6 `rgba()` calls, but the same anti-pattern (appending 2-char hex alpha to a theme color string) survived in 7 files. M2 was narrower than intended. The `+ '40'`/`+ '30'` syntax is only valid when `colors.primary` is in `#rrggbb` form — if any token migrates to `oklch()` output from `lib/_core/theme.ts:54`, these silently break.
- **Fix effort:** S
- **Suggested fix:** Add a `withAlpha(color, alpha)` helper to `lib/_core/theme.ts`; replace all string-concat sites.

### WR-5: 3 hardcoded `rgba()` still in app/tabs/index.tsx styles
- **File:** `app/tabs/index.tsx:38, 50, 56`
- **Category:** UI-UX (theming)
- **Severity:** WARNING
- **Description:** M2 left these `rgba(15,14,23,...)` calls in the dark-mode-scrim `withOpacity` fallback path. New finding: a `withOpacity` helper is defined inline (lines 36-55) that accepts a hex + alpha — the helper exists but the **StyleSheet** definitions (line 38, 50, 56) still hardcode rgba.
- **Fix effort:** S
- **Suggested fix:** Move `withOpacity` to `lib/_core/theme.ts` and use it for both the inline styles and the StyleSheet.

### WR-6: AudioPlayerService `fadeInterval` not cleared in `stop()`
- **File:** `lib/audio-player-service.ts:255-290`
- **Category:** Code-Quality (subtle audio bug)
- **Severity:** WARNING
- **Description:** `fadeVolume` (line 256 calls `_clearFade(trackId)`), but `stop()` (line ~140) only clears via `_clearFade` if the method is invoked. If `play()` is called while a fade is in flight on the same `trackId`, the new `play` does `await this.stop(trackId)` (line 56) which should clear. Verified: the `play` path looks correct, but `_clearFade` itself (searched via grep) wasn't found in this file — possibly a missing helper or inlined. Risk: stale interval runs after track removed.
- **Fix effort:** S
- **Suggested fix:** Verify `_clearFade` exists; ensure it clears `fadeInterval` and `fadeOnDone`. Add a unit test for fade-then-stop race.

### WR-7: `useTypewriter` doesn't respond to speed change mid-typing
- **File:** `hooks/useTypewriter.ts:14-31`
- **Category:** UI-UX
- **Severity:** WARNING
- **Description:** `startTypewriter` has `textSpeed` in its dependency array, but the consumer (e.g., `useReaderAudio`-adjacent readers) calls `startTypewriter` only on text-change. When the user adjusts text speed in settings mid-typing, the running interval keeps using the old `delay`. Either restart the typewriter on speed change, or document the behavior.
- **Fix effort:** S
- **Suggested fix:** Capture `delay` in a ref that the interval reads, so a speed change takes effect on the next char tick.

## Security Findings

### S-1: Strong — URI / path / SVG / data: filtering
- **Files:** `lib/story-validator.ts`, `lib/asset-resolver.ts:78-82`
- **Note:** `isSafeUri()` correctly blocks `javascript:`, `data:`, `vbscript:`, `..`, `\0`. `isSafeDataUri()` blocks `data:image/svg+xml` (XSS vector). `isPathSafe()` blocks filename dangerous chars. Solid foundation.

### S-2: OAuth state validation + PKCE-style flow
- **File:** `lib/_core/auth.ts:33-70`
- **Note:** `generateOAuthState()` uses `crypto.getRandomValues(Uint32Array(8))` — cryptographically secure. State is single-use (cleared after validation). Good.

### S-3: Session token isolation
- **File:** `lib/_core/auth.ts:72-115`
- **Note:** Web uses cookie-based auth (no token storage); native uses `expo-secure-store`. No token logged in any non-`api.ts:153` location (the new CR-2 finding). Bearer header correctly added on native only.

### S-4: `lib/_core/api.ts:96-98` correctly redacts Set-Cookie values
- Sanitized as `=***` before logging. Pattern is sound; consider applying the same to `authUserInfo` in callback.tsx.

### S-5: No `eval`, `dangerouslySetInnerHTML`, `__proto__` use in production code
- `grep` returned only `lib/engine/useSceneExecutor.ts:28` (`RESERVED_KEYS = new Set(['__proto__', 'constructor', 'prototype'])`) — used defensively to block variable names. Correct pattern.

### S-6: `audio-player-service.ts:30` is actually guarded
- The verification report flagged it as unguarded. On inspection, `logDebug` (line 28-31) has `if (!__DEV__) return;` guard on line 29. Same for `useReaderAudio.ts:101-104`. No fix needed.
