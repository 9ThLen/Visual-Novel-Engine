<objective>
Implement the authoring UX model: `Маша:` becomes a compact colored inline token, creates a character on first use, opens controls only once automatically, and then reuses the character's authoring defaults.
</objective>

<tasks>

<task type="auto">
  <name>Represent dialogue speaker as inline token metadata</name>
  <files>lib/document-editor/types.ts lib/document-editor/document-scene.ts</files>
  <action>
    Extend dialogue authoring data with enough metadata to render a speaker token:
    `characterId`, stable `speakerName`, `spriteId`, token color, and a flag that
    tells the editor whether first-use controls should open.

    Do not create a large visible technical block for ordinary dialogue authoring.
  </action>
  <acceptance_criteria>
    - `Маша:` remains a dialogue line in authoring.
    - Technical `character` steps are generated later, not shown as primary UI.
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Create or reuse characters from typed speaker names</name>
  <files>lib/document-editor/document-scene.ts lib/vn-plate-editor/scene-normalizer.ts stores/use-app-store.ts</files>
  <action>
    When normalizing or saving Plate document content, parse speaker names only
    from dialogue-leading speaker syntax. Avoid false positives for URLs and
    narration with internal colons. Resolve normalized speaker names against
    `characterLibraries[storyId]`. If absent, create a character with a
    persistent color and empty sprites, then emit a transient first-use
    `openCharacterPopover` command for that token.

    Existing characters must reuse existing color and authoring defaults.
  </action>
  <acceptance_criteria>
    - `Маша:`, `маша:`, and whitespace variants do not create duplicates.
    - First-use menu opens only for newly created characters in the current
      editor session and is not persisted.
    - URLs and narration colons are not parsed as speaker tokens.
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Render compact colored token</name>
  <files>lib/vn-plate-editor/embedded-renderers.ts lib/vn-plate-editor/embedded-script.ts</files>
  <action>
    Render speaker names as inline rectangular tokens with the character's
    persistent color. The token must occupy inline text flow and avoid increasing
    line height beyond normal dialogue text.

    Token DOM contract:
    `<span class="speaker-token" data-character-id="..." data-block-id="...">Маша:</span>`.

    Update `blockToHtml` for dialogue blocks so existing document dialogue
    renders the token from `block.characterId`, `block.speakerName`, and the
    character color. The outer paragraph and token must carry escaped
    `data-character-id` when available.

    Update `transformDialogueIfNeeded` so typed speaker text resolves against
    the WebView payload's characters case-insensitively:
    - found character: set `data-character-id` immediately
    - new character: emit creation/open-popover bridge event
    - after bridge callback: patch the DOM token with the created `characterId`

    Clicking the token sends an explicit bridge message:
    `postMessage({ type: 'openCharacterPopover', characterId, blockId })`.
    Controls must not be visible otherwise.

    Accessibility contract:
    - token is reachable by keyboard
    - role/label communicate "edit character {name}"
    - Enter/Space opens controls
    - screen readers can still read the dialogue line coherently
  </action>
  <acceptance_criteria>
    - Token is compact and inline.
    - Existing dialogue rehydrates with `data-character-id`.
    - Newly typed existing character names resolve to existing IDs.
    - Controls are hidden until token click.
    - Keyboard editing of dialogue text still works.
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Implement character popover controls</name>
  <files>lib/vn-plate-editor/embedded-script.ts</files>
  <action>
    Add controls for:
    - character display name
    - color
    - upload sprite
    - user-provided sprite name
    - rename/delete sprite
    - choose current authoring sprite
    - choose authoring position
    - focus-on-speak settings

    Changing the selected sprite updates the current dialogue token and the
    character's `authoring.currentSpriteId` for future new dialogue lines.

    On phone/native, use a bottom-sheet or full-width panel fallback if a popover
    cannot fit. Do not rely on tiny hover-only controls.

    Sprite upload must validate:
    - image type
    - file size
    - image dimensions
    - URI safety
    - duplicate names within the same character, with an explicit rename or
      auto-suffix rule

    Sprite deletion removes only the character sprite record unless the existing
    media library provides safe reference-aware physical deletion.

    Handle edge cases:
    - no sprites: show upload-first state, not an empty picker
    - missing selected sprite: fallback to default and show warning
    - rename character: update all tokens with the same `characterId`
    - delete character: show reference count and default to non-destructive
      cancel/disable behavior; if deletion proceeds, degrade existing tokens to
      plain speaker text or an "Unknown Character" warning state

    All visible labels, warnings, errors, and accessibility labels must go
    through `lib/translations.ts`.
  </action>
  <acceptance_criteria>
    - Sprites are only those uploaded for the selected character.
    - Sprite names are user-editable.
    - Existing character tokens do not auto-open controls.
    - Mobile has a usable non-hover control surface.
    - Deleting a sprite does not physically delete shared assets unsafely.
    - Duplicate sprite names are scoped per character.
    - Referenced characters cannot be deleted without an explicit warning path.
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Define undo/save semantics for character library edits</name>
  <files>lib/vn-plate-editor/embedded-script.ts stores/use-app-store.ts</files>
  <action>
    Produce an explicit implementation decision in this plan's output before
    changing behavior:
    - either integrate sprite/name/color changes with the editor undo stack
    - or treat them as explicit character-library mutations applied through save
      actions and documented as out-of-band from text Ctrl+Z

    If choosing editor undo integration, this task must ship the integration PR
    with tests. If choosing out-of-band save actions, this task must ship the
    user-visible save/retry behavior plus documentation of Ctrl+Z boundaries.

    In both cases, text undo must not leave tokens referencing missing or stale
    library state without a visible warning.

    Handle upload/save failure: if sprite upload or library mutation succeeds but
    scene save fails, show retry/recovery state and avoid silently pointing tokens
    at unsaved state.
  </action>
  <acceptance_criteria>
    - The selected undo model is documented in the implementation notes.
    - The implementation matches that selected undo model; no placeholder
      "decide later" state remains.
    - Ctrl+Z behavior is predictable.
    - Library changes are not silently lost on save.
    - Upload/save failure has visible recovery behavior.
  </acceptance_criteria>
</task>
