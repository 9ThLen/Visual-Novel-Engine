<objective>
Build the canonical save pipeline that turns inline dialogue authoring into internal character timeline steps without exposing those steps as large editor blocks.
</objective>

<tasks>

<task type="auto">
  <name>Generate character steps per scene</name>
  <files>lib/document-editor/document-scene.ts</files>
  <action>
    Rewrite the dialogue-to-timeline section to rebuild generated character
    steps from authoring blocks on each save. Do not append generated steps to an
    existing generated timeline; derive them from the current authoring content.

    Scan each scene top to bottom. Each scene starts from a clean character
    visibility state; do not inherit state from previous scenes or other story
    branches. Only explicit source timeline metadata that is already part of
    the current scene may override this. Track:
    - `visibleCharacters`
    - `currentSpriteByCharacter`
    - `currentPositionByCharacter`

    For each dialogue speaker:
    - if not visible, insert `character show`
    - if visible but sprite changed, insert `character change_sprite`
    - if position changed, insert `character move`
    - then insert the `dialogue` step

    Do not insert repeated `show` for later dialogue by an already visible
    character in the same scene. If all dialogue for a character was deleted, do
    not generate orphan character steps for that character.

    Generated `character show/move/change_sprite` steps must remain non-yielding
    in the executor, so the reader advances through them without pausing before
    the dialogue step. These inserted steps must not affect `currentEntryIndex`
    inside dialogue blocks.

    Preserve explicit technical character steps in timeline order. If explicit
    and generated character state conflict, the later step in timeline order
    wins. Generated steps must never reorder explicit steps.

    Dialogue lines must use a concrete `spriteId` snapshot. Updating
    `character.authoring.currentSpriteId` affects future new dialogue lines only,
    unless a separate explicit bulk-apply action is added later.

    `DialogueEntry.characterId` must always be the stable character library ID
    after Phase 17. Do not fall back to storing `speakerName` as `characterId`
    for new saves. Legacy rows without IDs should be migrated/resolved during
    document normalization before timeline generation.
  </action>
  <acceptance_criteria>
    - First speaker line creates a show step.
    - Later same-speaker lines do not replay entrance.
    - Sprite changes create explicit change steps.
    - Separate scenes do not inherit visible characters from each other.
    - Reader step flow skips generated character steps seamlessly.
    - Explicit technical character steps override generated state by timeline order.
    - Changing current sprite does not rewrite old dialogue lines unexpectedly.
    - Saved dialogue entries store library IDs, not display names, in `characterId`.
  </acceptance_criteria>
</task>

<task type="auto">
  <name>Move speaker focus into dialogue data</name>
  <files>lib/engine/types.ts lib/document-editor/document-scene.ts lib/reader-runtime.ts</files>
  <action>
    Add optional `speakerFocus` to `DialogueBlockData` and populate it from the
    character authoring focus settings. Do not create a `speak` character action.
    Existing dialogue steps without `speakerFocus` remain valid and serialize as
    undefined/no focus.

    Also implement the speaker display-name contract. Preferred path:
    - add optional `speakerName` to `DialogueEntry` as a display snapshot
    - keep `characterId` as the stable library ID
    - update `sceneRecordToDocumentScene` and `documentSceneToTimeline` so
      `speakerName`, `characterId`, and `spriteId` roundtrip without semantic
      drift

    If the implementation chooses render-time lookup instead, update every
    reader display helper that formats dialogue text to resolve `characterId`
    through `characterLibraries[storyId]`.

    In both cases, `getTimelineDisplayPages` must not show internal IDs such as
    `char_abc123` as speaker labels.
  </action>
  <acceptance_criteria>
    - Speaker focus serializes with dialogue.
    - Character steps remain limited to scene state actions.
    - Reader-visible dialogue labels use character display names, not IDs.
    - `sceneRecordToDocumentScene` preserves character IDs through `sourceStep`.
  </acceptance_criteria>
</task>
