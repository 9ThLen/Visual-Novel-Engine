# Plan 07.6-01: Fix Race Condition and Related Issues in useSceneExecutor

## Objective

Fix the confirmed race condition in `processNext` and resolve all critical/warning findings from the code review (07.5-REVIEW.md).

## Background

The `useSceneExecutor` hook uses `setTimeout(() => processNext(), 0)` inside `setSceneState` updater to process non-yielding steps recursively. When the `timeline` prop changes while a timeout is pending, the stale callback reads a reset `internalIndexRef` â€” causing duplicate step execution, inconsistent `sceneState`, and potential infinite loops.

3 critical, 4 warning, and 3 info findings were identified in the review.

## Tasks

### Task 1: Replace setTimeout with Synchronous Loop in processNext

<read_first>
- lib/engine/useSceneExecutor.ts (entire file)
- lib/engine/types.ts (TimelineStep, SceneState types)
- lib/engine/conditionUtils.ts (createEmptySceneState)
</read_first>

<action>
Replace the `processNext` function (lines 175-212) to use a synchronous `while` loop inside `setSceneState` instead of recursive `setTimeout`.

Current pattern:
```
processNext reads idx from internalIndexRef
  setSceneState(prev => {
    executeStep(step, prev) â†’ if 'continue', idx++, setTimeout(processNext, 0)
    if 'halt', set state setters inside updater
  })
```

New pattern:
```
processNext:
  setSceneState(prev => {
    while idx < steps.length:
      executeStep(step, state) â†’ if 'halt', update refs, return state
      if 'continue', idx++, continue loop
    return final state
  })
  // All setIsTyping/setCanAdvance/setCurrentStepIndex OUTSIDE updater
```

Key design:
1. Remove `setTimeout(() => processNext(), 0)` on line 205 â€” no more pending callbacks
2. Move all `setIsTyping`, `setCanAdvance`, `setCurrentStepIndex` calls to AFTER `setSceneState` returns
3. Track `haltType` via a local variable set inside the updater, used outside it
4. Update `internalIndexRef.current` only outside the updater (or at the very end inside it)
</action>

<acceptance_criteria>
- The `setTimeout(() => processNext(), 0)` call is eliminated
- No state setters (`setIsTyping`, `setCanAdvance`, `setCurrentStepIndex`) are called inside the `setSceneState(prev => ...)` updater function body
- All non-yielding steps in a row execute in a single synchronous pass
- Text/dialogue blocks still set `isTyping=true` and yield correctly
- Choice blocks still set `currentChoices` and halt
- `npm run check` passes (no TypeScript errors)
</acceptance_criteria>

### Task 3: Add try/finally to advanceGuardRef

<read_first>
- lib/engine/useSceneExecutor.ts (lines 228-257)
</read_first>

<action>
Wrap the body of `advance` between `advanceGuardRef.current = true` and its final reset in a `try/finally` block to prevent permanent deadlock on exception.

Current:
```
advanceGuardRef.current = true;
if (isTyping) { ... advanceGuardRef.current = false; return; }
if (sceneState.currentChoices) { ... advanceGuardRef.current = false; return; }
// ... more returns ...
advanceGuardRef.current = false;
```

New:
```
if (advanceGuardRef.current) return;
advanceGuardRef.current = true;
try {
  // all existing logic â€” remove every `advanceGuardRef.current = false; return;`
  // and replace each with just `return;`
} finally {
  advanceGuardRef.current = false;
}
```
</action>

<acceptance_criteria>
- `advanceGuardRef.current = false` is only written in ONE place â€” the `finally` block
- All early returns inside `advance` no longer explicitly reset `advanceGuardRef`
- If any code inside `advance` throws, `advanceGuardRef` is still reset to `false`
- `npm run check` passes
</acceptance_criteria>
