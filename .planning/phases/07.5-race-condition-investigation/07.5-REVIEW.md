# Phase 07.5: Code Review Report — Race Condition Investigation

**Reviewed:** 2026-05-29T14:59:00Z
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Reviewed `lib/engine/useSceneExecutor.ts` (289 lines) — the React hook that drives visual novel scene execution. The investigation confirmed the race condition in `processNext` caused by `setTimeout(() => processNext(), 0)` for recursive non-yielding step processing. Additionally, discovered several related issues: state setters called inside a state updater function (React anti-pattern), stale closure problems in `advance`/`selectChoice`, and an effect dependency on a reference-type value that can cause runaway re-execution.

**3 critical**, **4 warning**, and **3 info** findings.

---

## Critical Issues

### CR-01: Race Condition — Duplicate Step Execution via Stale `setTimeout`

**File:** `lib/engine/useSceneExecutor.ts:175-212`
**Issue:** The `processNext` function uses `setTimeout(() => processNext(), 0)` (line 205) inside the `setSceneState` updater to recursively process non-yielding steps. When the `timeline` prop changes while a `setTimeout` is pending, the stale callback executes with an `internalIndexRef.current` that was reset to 0 by the new `useEffect` run (line 215), causing the first step of the new timeline to be **executed twice**.

**Full trace of the race:**

```
1. Timeline A = [stepA0_nonyield] loads
2. useEffect resets internalIndexRef = 0, calls processNext()
3. processNext: idx=0, executeStep(stepA0) → 'continue'
4.   setSceneState updater: internalIndexRef = 1
5.   setTimeout(processNext, 0) queued  ← TIMEOUT_A
6. Timeline B = [stepB0_halt, stepB1] arrives (prop change)
7. Component re-renders → useEffect re-fires
8. Effect: internalIndexRef = 0, sceneState = empty, processNext()
9. processNext (new call): idx=0, executeStep(stepB0) → 'halt'
10.  internalIndexRef stays 0, isHaltedRef = true
11. TIMEOUT_A (from step 5) fires — this is the STALE callback
12. processNext reads internalIndexRef.current = 0  ← was reset by step 8
13. processNext reads timelineRef.current = [stepB0, stepB1]
14. executeStep(timeline[0], prev) = executeStep(stepB0, currentState)
15. ⚠️ stepB0 is executed AGAIN on top of state that already has stepB0 applied!
```

For a `dialogue` step, this duplicates the history entry. For a `variable` step, it applies the operation twice. For a `transition` step, it double-sets the transition target.

**Fix:** Replace the recursive `setTimeout` pattern with a **synchronous loop** inside `setSceneState` that processes all non-yielding steps in one pass, then yields at the first halt. This eliminates the race entirely because no pending callback can exist when the timeline changes.

```typescript
const processNext = useCallback(() => {
  const steps = timelineRef.current;
  let idx = internalIndexRef.current;

  setSceneState((prev) => {
    let currentState = prev;

    while (idx < steps.length) {
      const step = steps[idx];
      const { nextState, result } = executeStep(step, currentState);
      currentState = nextState;

      if (result === 'halt') {
        isHaltedRef.current = true;
        setCanAdvance(step.blockType !== 'choice');
        if (step.blockType === 'text' || step.blockType === 'dialogue') {
          setIsTyping(true);
          setCanAdvance(true);
        }
        setCurrentStepIndex(idx);
        internalIndexRef.current = idx;
        return currentState;
      }
      idx++;
    }

    // Reached end — all steps executed synchronously
    setCanAdvance(false);
    setCurrentStepIndex(idx);
    internalIndexRef.current = idx;
    return currentState;
  });
}, [executeStep]);
```

### CR-02: State Setters Called Inside `setSceneState` Updater Function

**File:** `lib/engine/useSceneExecutor.ts:186-211`
**Issue:** The updater function passed to `setSceneState` (line 186) calls **multiple other state setters** inside its body:

| Line | Setter | Context |
|------|--------|---------|
| 190 | `setIsTyping(true)` | text/dialogue branch |
| 191 | `setCanAdvance(true)` | text/dialogue branch |
| 196 | `setCanAdvance(...)` | halt branch |
| 197 | `setCurrentStepIndex(idx)` | halt branch |
| 202 | `setCurrentStepIndex(idx + 1)` | continue branch |
| 204-208 | `setTimeout` + `setCanAdvance` | continue branch |

React's state updater functions (the `(prev) => next` pattern) are **supposed to be pure functions** that only compute and return the next state. Calling other state setters inside them is explicitly warned against — it can cause:

- The inner setters to be called with stale values
- State update ordering to be unpredictable (React may flush them in unexpected order)
- With concurrent features (React 18+), potential tearing where different render cycles see different partial state

**Fix:** Refactor `processNext` into two phases: (1) compute the next state and determine the halt/continue decision purely inside the updater, then (2) set all derived boolean/scalar state **after** the updater returns. The `advance` function serves as the model — it calls `setSceneState` first, then sets `isHaltedRef`, `canAdvance`, etc. after.

```typescript
const processNext = useCallback(() => {
  const steps = timelineRef.current;
  let nextIndex: number = 0;
  let halted = false;
  let haltType: string | undefined;

  setSceneState((prev) => {
    let currentState = prev;
    let idx = internalIndexRef.current;

    while (idx < steps.length) {
      const step = steps[idx];
      const { nextState, result } = executeStep(step, currentState);

      if (result === 'halt') {
        halted = true;
        haltType = step.blockType;
        internalIndexRef.current = idx;
        return nextState;
      }
      currentState = nextState;
      idx++;
    }

    halted = false;
    internalIndexRef.current = idx;
    return currentState;
  });

  // All derived state is set OUTSIDE the updater
  nextIndex = internalIndexRef.current;
  setCurrentStepIndex(nextIndex);

  if (halted && haltType) {
    isHaltedRef.current = true;
    if (haltType === 'text' || haltType === 'dialogue') {
      setIsTyping(true);
      setCanAdvance(true);
    } else {
      setCanAdvance(haltType !== 'choice');
    }
  } else {
    isHaltedRef.current = false;
    setCanAdvance(false);
  }
}, [executeStep]);
```

**Risk of the current code in React 19 / concurrent mode:** Medium-to-high. While React 18 batches these calls in most scenarios, calling setters inside updaters violates React's contract and may break with future concurrent features.

### CR-03: Effect Dependency on `options?.initialVariables` Causes Runaway Re-Execution

**File:** `lib/engine/useSceneExecutor.ts:226`
**Issue:** The `useEffect` at line 214 depends on `options?.initialVariables`:

```typescript
useEffect(() => {
  // reset all state and restart processing
  ...
}, [timeline, options?.initialVariables, processNext]);
```

`options?.initialVariables` is a **reference-type value** (an object `Record<string, string | number | boolean>`). If the parent component passes an inline object:

```typescript
useSceneExecutor(timeline, { initialVariables: { hp: '100' } })
```

…then **every render creates a new object**, this effect dependency changes every render, and the effect fires on every render cycle. This causes the scene state to be repeatedly reset and `processNext` to restart in an infinite loop.

Even if the parent memoizes the object (`useMemo`), any parent re-render that recreates `options` will reset the entire scene — a severe correctness bug.

**Fix:** Remove `options?.initialVariables` from the dependency array because the effect already reads `options` directly. Instead, use a ref to store the initial variables for comparison, or accept that the effect only needs to react to `timeline` changes:

```typescript
const initialVariablesRef = useRef(options?.initialVariables);
initialVariablesRef.current = options?.initialVariables;

useEffect(() => {
  internalIndexRef.current = 0;
  isHaltedRef.current = false;
  advanceGuardRef.current = false;
  setSceneState({
    ...createEmptySceneState(),
    variables: initialVariablesRef.current ?? {},
  });
  setCurrentStepIndex(0);
  setIsTyping(false);
  setCanAdvance(false);
  processNext();
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, [timeline]);
```

Alternatively, extract `initialVariables` via `useMemo` inside the hook or use a stable serialized key. The key insight: the effect resets state whenever timeline changes; resetting on `initialVariables` changes is a secondary concern that should be handled differently (e.g., via a dedicated key prop).

**Risk:** High. If the parent passes `options` inline (which is the natural usage pattern), this causes an infinite effect loop that repeatedly resets scene state, making the editor/preview non-functional.

## Warnings

### WR-01: Stale Closure in `advance` — `sceneState.currentChoices` Captured at Creation Time

**File:** `lib/engine/useSceneExecutor.ts:228-257`
**Issue:** The `advance` callback uses `useCallback` with dependency `[isTyping, sceneState.currentChoices, processNext]` (line 257). The value of `sceneState.currentChoices` is capured from the render when the callback was created. If a user tap event handler was registered with an old `advance` reference (before choices were set), the guard at line 241 will **not fire**, allowing an advance through a choice block:

```typescript
if (sceneState.currentChoices) {
  advanceGuardRef.current = false;
  return;  // ← this check uses stale value
}
```

Similarly, if `advance` was registered when choices existed but they were already cleared (e.g., by `selectChoice`), `advance` will incorrectly skip this guard and proceed to `isHaltedRef.current` check.

**Fix:** Use a ref for the current choices check, or ensure `advance` is re-registered after every state change (current approach with dep array, but event handlers getting stale references is still possible):

```typescript
const choicesRef = useRef(sceneState.currentChoices);
choicesRef.current = sceneState.currentChoices;

// Inside advance:
if (choicesRef.current) { ... }
```

### WR-02: Stale Closure in `selectChoice` — Same Pattern

**File:** `lib/engine/useSceneExecutor.ts:259-278`
**Issue:** Same stale closure pattern as WR-01. `selectChoice` depends on `[sceneState.currentChoices]` (line 278). If `sceneState.currentChoices` is an empty array `[]`, the guard at line 260 `if (!currentChoices) return;` does NOT catch it (empty array is truthy). The subsequent `find` at line 263 will return `undefined`, and the function returns early at line 264. This is a no-op but silently swallows the selection.

If the currentChoices array is stale (empty vs null vs populated), the function may return early or proceed incorrectly.

**Fix:** Same ref-based approach as WR-01. Additionally, add an explicit check for empty choices array:

```typescript
const currentChoices = choicesRef.current;
if (!currentChoices || currentChoices.length === 0) return;
```

### WR-03: `advanceGuardRef` Can Get Permanently Stuck

**File:** `lib/engine/useSceneExecutor.ts:229,237,242,248,255`
**Issue:** The `advanceGuardRef` pattern:

```typescript
if (advanceGuardRef.current) return;
advanceGuardRef.current = true;
// ... logic with 4 return paths ...
advanceGuardRef.current = false;  // last return path at line 255
```

If any code path between lines 229-255 throws an **unhandled exception**, `advanceGuardRef.current` remains `true` permanently, deadlocking all future `advance` calls. The ref is only reset to `false` in the `useEffect` (line 217), which requires a `timeline` prop change to unstick it.

**Fix:** Use a `try/finally` block:

```typescript
const advance = useCallback(() => {
  if (advanceGuardRef.current) return;
  advanceGuardRef.current = true;
  try {
    // ... all existing logic ...
  } finally {
    advanceGuardRef.current = false;
  }
}, [isTyping, sceneState.currentChoices, processNext]);
```

### WR-04: `selectChoice` Does Not Call `processNext` — Steps After Choice Are Orphaned

**File:** `lib/engine/useSceneExecutor.ts:259-278`
**Issue:** When a choice is selected (line 259), the function:

1. Sets `isHaltedRef.current = false` (line 274)
2. Increments `internalIndexRef.current` (line 276)
3. Sets `isTransitioning = true` with a `transitionTarget` (lines 270-271)
4. Does **NOT** call `processNext` (line 278 returns)

If the timeline has steps after the choice block (e.g., `choice → background → dialogue`), those steps are **never executed** because `processNext` is not called here, and `advance` will short-circuit at `isHaltedRef.current == false` (line 246-248). The only mechanism to proceed is an external scene transition that reloads with a new timeline.

This is a design defect: if a choice's `targetSceneId` is the same scene or undefined, remaining steps are orphaned. The consumer must be aware that choice always terminates processing regardless of what follows in the timeline.

**Fix:** Either (a) call `processNext()` after the state updates when there is no transition target, or (b) document this behavior clearly with a comment at the `break` in the `case 'choice'` handler (line 128-131) and at the choice block handler in `processNext` (line 194-198) noting that "choice always halts and the caller must handle the transition; steps after choice in the same timeline are not executed."

## Info

### IN-01: `setCanAdvance(true)` in Typing Branch is a No-Op

**File:** `lib/engine/useSceneExecutor.ts:236`
**Issue:** In the `advance` function's typing branch (line 232-239):

```typescript
if (isTyping) {
  setIsTyping(false);
  setCanAdvance(true);  // ← canAdvance is already true at this point
  advanceGuardRef.current = false;
  return;
}
```

`canAdvance` was already set to `true` when the text/dialogue step was first encountered (line 191). This `setCanAdvance(true)` is a no-op that adds confusion about state machine transitions. It should either be removed, or the comment should explain the intent. If the intent is "the next tap after typing still advances," the variable name or state machine design should be revisited.

**Fix:** Remove line 236, or add a clear comment: `// Keep canAdvance=true so the next tap triggers the real advance`.

### IN-02: `dialogueHistory` Stores `characterId` as `characterName`

**File:** `lib/engine/useSceneExecutor.ts:119`
**Issue:** The dialogue entry is created with:

```typescript
characterName: entry.characterId,
```

This stores the character's ID in the `characterName` field instead of looking up the display name. Downstream UI that renders `characterName` will display the raw ID (e.g., "narrator" instead of "Narrator" or a localized name). This is a data integrity issue that degrades the user-facing experience.

**Fix:** If `characterId` is the best available name, add a clarifying comment. If a proper name field exists on the entry data, use it: `characterName: entry.characterName ?? entry.characterId`.

### IN-03: `isComplete` Derives from Ref with No Explicit Re-Render Trigger

**File:** `lib/engine/useSceneExecutor.ts:69`
**Issue:** The `isComplete` return value is computed from `internalIndexRef.current >= timeline.length`. This ref is updated inside `setSceneState` updaters (lines 201, etc.), but if `processNext` takes the early-return path at line 179 (`setCanAdvance(false)` only), the component re-renders due to `setCanAdvance`, which causes `isComplete` to recompute correctly. So it works — but fragilely. Any future refactoring that removes the `setCanAdvance` call would silently break `isComplete` without a compile-time warning.

**Fix:** Either derive `isComplete` from a real state variable, or add a comment:

```typescript
// Derives from ref but correctly triggers re-render via setCanAdvance(false) at end
const isComplete = internalIndexRef.current >= timeline.length;
```
